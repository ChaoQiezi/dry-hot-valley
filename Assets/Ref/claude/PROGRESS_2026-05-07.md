## 2. 大渡河方法 A: 严格河谷 polygon 内 3 km VAI

### 2.1 设计

在严格 `Daduhe_valley.shp` polygon 内, 像素级地重算 3 km 网格 VAI:

```
[Daduhe_valley.shp polygon] ──→ 10m 像素级 mask
                                 ↓
                        [NDVI/DEM/windward_leeward _region 输入]
                                 ↓
              对每个 3 km cell(300×300 像素):
                只用 polygon 内像素 → 分迎风/背风 → 计算 VAI
                                 ↓
                  VAI_3km_strict.tif + 7 个辅助栅格
                                 ↓
                  cell-level LOWESS + bootstrap CI 阈值分析
```

关键差异点(与既有 `VAI_spatial_distribution.py`):

1. **像素级空间约束**: `valid_mask = (poly_block == 1) & ...`
2. **修复 MIN_PIXEL_THRESHOLD bug**: 既有脚本第 69 行定义 `MIN_PIXEL_THRESHOLD = 15` 但从未实际使用
   (line 162 计算 `n_ww, n_lw` 后直接 `np.mean()` 无判断). 新脚本严格应用 `MIN_PIXEL_THRESHOLD = 50`.
3. **输出 valley_fraction 栅格**: 用于后续灵活筛选边界 cell.

### 2.2 代码与输出

- 代码: `F:\PyProJect\dry_hot_valley\Src\valley_analysis\Daduhe\geo_analysis\VAI_strict_valley_3km.py`
- 阈值脚本: `F:\PyProJect\dry_hot_valley\Src\valley_analysis\Daduhe\geo_analysis\VAI_strict_valley_threshold.py`
- 输出栅格: `E:\GeoProjects\dry_hot_valley\Daduhe\VAI\VAI_3km_strict.tif`(及 7 个辅助栅格)
- 输出表: `E:\GeoProjects\dry_hot_valley\Daduhe\Result\Table\altitude\VAI_strict_*.csv`
- 图: `E:\GeoProjects\dry_hot_valley\Daduhe\Result\Chart\altitude\VAI_strict_threshold.png`

### 2.3 结果

| 项目 | 数值 |
|---|---|
| 有效 3 km 网格 | 226 (frac ≥ 0.10) |
| DEM 范围 | 728–3403 m |
| 相对河道高差范围 | -318 至 +1137 m |
| VAI 中位数 | +15.2% (p10=+2.7%, p90=+33.0%) |
| t 检验 p<0.05 占比 | 98.2% |
| VAI < 0 cells | 6.6% |
| ABS 反转(中位/CI) | 3263 m [3047, 3391] |
| ABS bootstrap 支持 | 293/500 = 58.6% |
| REL 反转(中位/CI) | 1000 m [874, 1129] |
| REL bootstrap 支持 | 321/500 = 64.2% |

### 2.4 与 `_region` 的诊断对比(同 cell 配对差分)

| valley_fraction 区间 | n cells | mean diff (strict-region) | 中位 \|diff\| | max \|diff\| |
|---|---:|---:|---:|---:|
| ≥ 0.95 (几乎全在 polygon 内) | 4 | -0.03% | **0.00%** | 0.10% |
| 0.80 – 0.95 | 18 | +0.32% | 0.88% | 10.0% |
| 0.50 – 0.80 | 70 | +2.55% | 4.10% | 14.6% |
| 0.30 – 0.50 | 71 | +3.24% | 6.04% | 38.4% |
| 0.10 – 0.30 (cell 大半在 polygon 外) | 63 | +3.36% | 5.82% | 109.2% |

**核心发现**:
- 完全在 polygon 内的 cell, 两版本 VAI 几乎完全相同(diff ≤ 0.1%) — 算法逻辑等价.
- 差异**仅来自边界 cell 用了哪些像素**: `_region` 把 polygon 外的山坡像素一起算, `_strict` 只用 polygon 内像素.
- 系统性 mean diff > 0: **polygon 外的山坡像素拉低了 region 版本的 VAI**.
  这与干热河谷物理直觉一致 — 严格河谷内迎风优势更强.

### 2.5 评价

- **优点**: 严格符合"干热河谷"的物理边界定义, 修复了既有 bug.
- **缺点**:
  1. **样本量不足**: 226 cells, 高海拔段(>3000 m)极稀疏, 导致 bootstrap 仅 58.6% 支持反转点.
  2. **依赖人为 polygon**: 不同来源/年份的"干热河谷"polygon 边界不一致, 难以跨河谷标准化.
  3. **岷江 polygon CRS 是 Albers 而非 UTM 47N** — 这个发现源于 plot 时 polygon 错位 bug, 已修复.

## 3. 大渡河方法 B: 河道 4 km buffer 内 3 km VAI

### 3.1 为什么选 4 km buffer

用户提议 3-5 km 自决, 选 4 km 的依据:
- 3 km 与 grid 同尺度, 边缘 cell 容易卡阈值;
- 5 km 横向 10 km, 可能吃到非河谷山坡;
- **4 km 横向 8 km**, 接近 Daduhe 实测河谷半宽, 是"半窄不宽"的中间值.

### 3.2 设计

```
[centerline_final.shp 中"大渡河干旱河谷"特征]
            ↓ 投影到栅格 CRS, shapely.LineString.union → buffer(4000m)
[10m 河道 4 km buffer 栅格 mask]
            ↓ 与方法 A 同一套流程
[VAI_3km_buffer.tif]
            ↓
[cell-level LOWESS + bootstrap CI]
```

  1. 样本量是 strict 的 ~2.4 倍.

## 4. 关键 bug 修复记录

### 4.1 `MIN_PIXEL_THRESHOLD` 未实际应用(既有 _region 主线的潜在 bug)

**症状**: 既有 `VAI_spatial_distribution.py` 文件 line 69 定义 `MIN_PIXEL_THRESHOLD = 15`,
line 162 计算 `n_ww, n_lw = len(...)` 但未使用; 直接进入 `np.mean()` 计算 VAI.

**影响**: `_region` 输出中可能存在仅靠几个像素就出 VAI 的网格, 数值噪声极大.

**新脚本中的处理**: `VAI_strict_valley_3km.py` 与 `VAI_buffer_valley_3km.py` 都严格应用
`MIN_PIXEL_THRESHOLD = 50`(更严格), 任一侧像素数 < 50 则该网格 VAI 输出为 NaN.

### 4.2 岷江 polygon 在 plot 中错位 bug(CRS 不一致)

**症状**: 用户用 ArcGIS 看 Minjiang VAI_strict_threshold.png 时, panel D 中 polygon 位置完全偏离 cells.

**Root cause**:
| 河谷 | shapefile CRS |
|---|---|
| Daduhe | UTM 47N ✓ |
| **Minjiang** | **Asia_North_Albers ✗** |
| Jinshajiang | UTM 47N ✓ |
| Yalongjiang | UTM 47N ✓ |

`load_polygon_xy()` 函数中我之前注释"已是 UTM 47N, 不需变换" — 对岷江不成立.
栅格在 EPSG:32647, polygon 在 Albers, 直接画图时 polygon 顶点落到完全不同的平面坐标.

**注**: VAI 计算脚本本身没问题 — 它用 `read_polygon_in_crs()` 做了变换, 像素级 mask 是对的, VAI 数值是对的. 仅 plot 出错.

**修复**: 改 `load_polygon_xy()` 为读 .prj → 与栅格 CRS 比较 → 不一致时用 pyproj 变换. 已应用到全部 4 河谷的 threshold 脚本.

### 4.3 TwoSlopeNorm 在岷江报错(用户已自行修复)

**症状**: 岷江 VAI 95.3% > 0, `np.percentile(VAI, 2)` ≥ 0, `TwoSlopeNorm(vmin=>=0, vcenter=0)` 报错.

**用户的修复**(在 Minjiang 的 threshold 脚本中):
```python
_vlo = np.nanpercentile(grid["VAI"], 2)
_vhi = np.nanpercentile(grid["VAI"], 98)
if _vlo >= 0:
    _vlo = -abs(_vhi) * 0.05
if _vhi <= 0:
    _vhi = abs(_vlo) * 0.05
norm = TwoSlopeNorm(vcenter=0, vmin=_vlo, vmax=_vhi)
```

**评价**: 修复合理, 避免报错; 仅副作用是色阶在四河之间不可比.
**论文出图建议**: 跨河谷比较时改用对称 `vmin=-vmax` 或固定 `±50%` 范围.
